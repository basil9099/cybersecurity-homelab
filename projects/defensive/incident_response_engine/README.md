# AEGIS - Automated Engagement & Guardian Incident System

**Incident Response Playbook Engine**

AEGIS is an automated incident response engine that executes predefined playbooks in response to security alerts. It maps every action to NIST 800-61 (Computer Security Incident Handling Guide) phases, providing structured, auditable, and repeatable incident response workflows.

---

## Architecture

```
aegis/
├── main.py                  # CLI entry point (demo, run, list modes)
├── engine/
│   ├── playbook_runner.py   # Core orchestrator: loads playbooks, executes steps
│   ├── action_executor.py   # Executes individual response actions (simulated or live)
│   └── alert_matcher.py     # Matches alerts to playbooks by type, severity, tags
├── models/
│   ├── alert.py             # Alert dataclass (type, severity, IPs, tags, raw data)
│   ├── playbook.py          # Playbook dataclass loaded from YAML definitions
│   └── response.py          # Execution result with timeline and evidence tracking
├── playbooks/
│   ├── brute_force.yaml     # SSH/RDP brute force response
│   ├── malware_detected.yaml# Malware containment and forensic collection
│   ├── data_exfiltration.yaml# Data exfil blocking and evidence preservation
│   └── unauthorized_access.yaml # Account lockdown and session review
├── reporting/
│   └── incident_report.py   # JSON and HTML report generation with NIST mapping
├── demo/
│   └── scenarios.py         # Pre-built scenarios simulating full response lifecycle
├── requirements.txt
└── README.md
```

### Data Flow

```
Alert (JSON/dict)
  │
  ▼
AlertMatcher ──► Selects best-matching Playbook (by type, severity, tags)
  │
  ▼
PlaybookRunner ──► Iterates through Playbook steps in sequence
  │
  ▼
ActionExecutor ──► Executes each action (simulated in demo mode)
  │                  ├── block_ip
  │                  ├── isolate_host
  │                  ├── collect_evidence
  │                  ├── notify / escalate
  │                  └── ... (13 action types)
  ▼
ResponseResult ──► Timeline, evidence, status, NIST phase mapping
  │
  ▼
IncidentReporter ──► JSON + HTML reports
```

## NIST 800-61 Phase Mapping

Every playbook step is mapped to a phase from the NIST 800-61 Incident Response Lifecycle:

| Phase | Description | Example Actions |
|-------|-------------|-----------------|
| **Preparation** | Readiness activities before an incident | Policy loading, tool initialization |
| **Detection** | Identifying and analyzing the incident | Log event, audit logs, alert correlation |
| **Containment** | Limiting the impact of the incident | Block IP, isolate host, disable account |
| **Eradication** | Removing the threat from the environment | Network scan, YARA rule sweep, malware removal |
| **Recovery** | Restoring systems to normal operation | Service restart, VLAN restoration |
| **Post-Incident** | Review and lessons learned | Notifications, escalation, report generation |

Reports generated by AEGIS include a NIST phase summary showing how many steps were executed in each phase, providing clear audit documentation.

## Playbook Format

Playbooks are defined in YAML with the following structure:

```yaml
playbook:
  name: "Playbook Display Name"
  description: "What this playbook does and when to use it."
  nist_phase: "containment"          # Primary NIST phase
  version: "1.0"
  author: "Your Team"
  tags: [network, malware]

  trigger_conditions:
    alert_types: [malware_detected]  # Which alert types activate this
    min_severity: high               # Minimum severity threshold
    required_tags: []                # All must be present on the alert
    any_tags: [malware, trojan]      # At least one must be present

  steps:
    - name: "Human-readable step name"
      action: block_ip               # Action type (see list below)
      parameters:                    # Action-specific parameters
        duration: "24h"
        firewall: "iptables"
      timeout: 60                    # Max execution time (seconds)
      rollback_action: log_event     # Action to run if this step fails
      rollback_parameters:
        message: "Rollback triggered"
      continue_on_failure: false     # Stop or continue on failure
      nist_phase: containment        # NIST phase for this step

  escalation_policy:
    notify_on_failure: true
    escalation_contacts: ["soc-lead@org.local"]
    escalation_threshold: 2          # Failures before escalation
    auto_escalate_severity: [critical]
```

### Supported Action Types

| Action | Description |
|--------|-------------|
| `log_event` | Log a security event to the incident record |
| `block_ip` | Add a firewall rule to block an IP address |
| `isolate_host` | Move a host to a quarantine VLAN |
| `collect_evidence` | Gather forensic artifacts (logs, disk images, samples) |
| `notify` | Send notifications via email or messaging |
| `escalate` | Escalate to higher-tier responders |
| `run_script` | Execute a response script on a target system |
| `disable_account` | Disable a user account in the directory service |
| `capture_traffic` | Capture network traffic (PCAP) for analysis |
| `scan_network` | Scan network ranges for indicators of compromise |
| `review_sessions` | Review and terminate active user sessions |
| `audit_logs` | Analyze logs for anomalies within a timeframe |
| `preserve_logs` | Copy logs to tamper-proof storage |

## Usage

### Prerequisites

```bash
pip install -r requirements.txt
```

Requires Python 3.10 or later.

### Run All Demo Scenarios

```bash
python main.py demo
```

This runs four scenarios (brute force, malware, data exfiltration, unauthorized access) with simulated actions and generates JSON and HTML reports in the `reports/` directory.

### Run a Specific Demo Scenario

```bash
python main.py demo brute_force
python main.py demo malware
python main.py demo exfiltration
python main.py demo unauthorized_access
```

### List Available Playbooks

```bash
python main.py list-playbooks
```

### List Demo Scenarios

```bash
python main.py list-scenarios
```

### Process a Custom Alert

```bash
# Inline JSON
python main.py run --alert '{"alert_type":"brute_force","severity":"high","source_ip":"10.0.0.1","dest_ip":"10.0.0.2","description":"Test alert","tags":["ssh","authentication"]}'

# From file
python main.py run --alert @my_alert.json
```

### Live Mode (Non-Simulated)

```bash
python main.py run --alert @alert.json --live
```

> **Warning:** Live mode will attempt to execute real actions. Use only in controlled environments with proper authorization.

---

## Reports

AEGIS generates two report formats after each playbook execution:

- **JSON** - Machine-readable, suitable for SIEM ingestion or API consumption
- **HTML** - Human-readable with a dark-themed UI showing timeline, evidence, NIST mapping, and execution summary

Reports are saved to the `reports/` directory with filenames based on the response ID (e.g., `incident_RESP-A1B2C3D4.html`).

## Ethical Disclaimer

AEGIS is designed for **authorized security operations, educational purposes, and controlled lab environments only**.

- **Authorization Required:** Only deploy and operate this tool in environments where you have explicit written authorization.
- **Lab Use:** This project is part of a cybersecurity homelab and is intended for learning and experimentation.
- **Demo Mode:** The default execution mode simulates all actions. No real network changes, firewall rules, or account modifications are made in demo mode.
- **Live Mode Risk:** Enabling live mode (`--live`) will attempt to execute real incident response actions. Ensure you have proper authorization, change management approval, and understand the implications before using live mode.
- **No Warranty:** This software is provided as-is for educational purposes. It is not a substitute for professional incident response tools or processes.
- **Responsible Use:** Always follow your organization's incident response policies, legal requirements, and ethical guidelines when handling security incidents.
